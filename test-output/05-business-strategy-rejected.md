# Test #5: 商业策略场景不适用 - 技能响应

## 用户输入
> 我们想做一款面向 Z 世代的短视频产品，目标是年底达到 1000 万 DAU，通过算法推荐和社交裂变实现增长，商业化路径是先免费后会员订阅

## 技能响应

该需求**不适合使用 EARS 语法**描述。

### 原因说明

EARS (Easy Approach to Requirements Syntax) 是一种用于描述**业务逻辑、状态流转、系统行为**的结构化需求语法。根据适用场景判断：

**✅ EARS 适用场景**：
- 复杂的业务逻辑与规则（支付状态流转、订单处理、积分计算）
- 后台系统交互与 API 规范
- 状态机相关需求
- 异常与容错处理

**❌ EARS 不适用场景**：
- 纯前端视觉与 UI/UX 动效
- **商业目标与市场策略** ← 您的需求属于此类
- 过于早期、发散性的想法

您的需求聚焦于：
- **商业目标**：年底达到 1000 万 DAU
- **增长策略**：算法推荐、社交裂变
- **商业模式**：先免费后会员订阅
- **目标用户**：Z 世代

这些属于**商业策略和产品规划**范畴，而非具体的系统行为或业务规则描述。

### 建议替代方案

对于商业策略和产品规划，建议使用以下文档类型：

1. **商业计划书 (Business Plan)**
   - 市场分析、竞争分析
   - 商业模式画布（Business Model Canvas）
   - 增长策略和关键指标（OKR/KPI）

2. **产品需求文档 (PRD) - 传统格式**
   - 产品愿景和目标
   - 用户画像和使用场景
   - 功能列表和优先级

3. **产品路线图 (Roadmap)**
   - 阶段性目标规划
   - 功能发布计划
   - 资源分配策略

4. **数据指标定义文档**
   - DAU/MAU 计算方法
   - 留存率、转化率定义
   - 数据埋点规范

### 如果您有业务逻辑需求...

如果您需要描述短视频产品的**业务逻辑**，例如：

| 适用 EARS 的场景 | 示例 |
|:---|:---|
| 内容推荐算法规则 | 当用户观看视频超过 30 秒时，系统应当记录兴趣标签 |
| 视频发布流程 | 当用户上传视频时，系统应当进行内容审核 |
| 用户等级体系 | 当用户粉丝数达到 1000 时，系统应当升级为 Lv.2 |
| 会员权限管理 | 当非会员用户尝试使用会员功能时，系统应当提示升级 |
| 内容审核状态机 | 当视频处于审核中状态时，系统应当仅对用户可见 |
| 异常情况处理 | 如果视频审核超时 24 小时，则系统应当转人工审核 |

这类需求非常适合使用 EARS 语法。请提供具体的业务规则描述，我将为您生成结构化的 PRD。
