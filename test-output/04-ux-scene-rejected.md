# Test #4: UI/UX 场景不适用 - 技能响应

## 用户输入
> 帮我设计一个社交 APP 的首页，要好看、有动效，用户刷起来爽，最好能用渐变色和卡片式布局

## 技能响应

该需求**不适合使用 EARS 语法**描述。

### 原因说明

EARS (Easy Approach to Requirements Syntax) 是一种用于描述**业务逻辑、状态流转、系统行为**的结构化需求语法。根据 EARS 的适用场景判断标准：

**✅ EARS 适用场景**：
- 复杂的业务逻辑与规则
- 后台系统交互与 API 规范
- 状态机相关需求
- 异常与容错处理

**❌ EARS 不适用场景**：
- 纯前端视觉与 UI/UX 动效
- 商业目标与市场策略
- 过于早期、发散性的想法

您的需求涉及"渐变色"、"卡片式布局"、"动效"等**视觉和交互设计**元素，属于 UI/UX 设计范畴，而非业务逻辑或系统行为描述。

### 建议替代方案

对于 UI/UX 设计需求，建议使用以下方法表达：

1. **设计稿/原型图**
   - 使用 Figma、Sketch、Axure 等工具绘制高保真设计稿
   - 标注配色方案、间距、字体等视觉规范

2. **交互说明文档**
   - 使用流程图描述用户操作路径
   - 用动画原型展示动效细节

3. **设计规范文档**
   - 定义设计系统（Design System）
   - 包含组件库、色彩系统、布局规范等

4. **用户故事 + 接受标准**
   - 描述用户目标和期望体验
   - 用 Gherkin 语法描述交互场景（Given-When-Then）

### 如果您有业务逻辑需求...

如果您需要描述社交 APP 的**业务逻辑**，例如：
- 内容推荐算法规则
- 用户关注/粉丝关系管理
- 内容审核流程
- 消息通知机制

这类需求非常适合使用 EARS 语法。请提供相关业务规则描述，我将为您生成结构化的 PRD。

---

**结论**：该测试用例验证技能能够正确识别 UI/UX 场景不适合 EARS，并给出合理的替代建议。
