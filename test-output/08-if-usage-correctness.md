# 库存管理系统 PRD

## 1. 文档信息
- **版本**: v1.0
- **日期**: 2026-04-12
- **作者**: AI Assistant
- **状态**: 已批准

## 2. 背景与目标
构建电商库存管理系统，支持库存扣减、库存预占、库存释放、库存告警等核心功能，妥善处理超卖、库存同步延迟、活动爆单等异常情况。

## 3. 全局名词定义 (Glossary)

| 术语 | 定义 | 取值范围/示例 |
|:---|:---|:---|
| **Inventory** | 商品库存记录 | - |
| **StockReservation** | 库存预占记录 | - |
| **Order** | 用户订单 | - |
| **InventoryOperation** | 库存操作类型 | [Deduct, Release, Reserve, Replenish] |
| **StockAlertLevel** | 库存告警级别 | [Normal, Low, Critical, OutOfStock] |
| **FlashSale** | 秒杀活动 | - |

## 4. 非功能性需求 (NFRs)

| Req ID | 模式 | 需求描述 |
|:---|:---|:---|
| NFR-001 | Ubiquitous | 系统应当保证库存扣减操作的原子性 |
| NFR-002 | Ubiquitous | 系统应当支持每秒至少 5000 次库存查询 |
| NFR-003 | Ubiquitous | 系统应当保证库存数据最终一致性 |
| NFR-004 | Ubiquitous | 系统应当在库存变更后 1 秒内同步到查询接口 |

## 5. 功能性需求 (EARS Requirements)

### 5.1 库存扣减模块

| Req ID | 模式 | 需求描述 |
|:---|:---|:---|
| REQ-001 | When | **When** 用户提交订单时，系统应当检查商品库存是否充足 |
| REQ-002 | If | **If** 订单提交时商品库存不足，则系统应当阻止订单创建并提示用户 |
| REQ-003 | When | **When** 订单支付成功时，系统应当正式扣减库存 |
| REQ-004 | If | **If** 库存扣减时发生并发冲突，则系统应当重试最多 3 次 |
| REQ-005 | If | **If** 重试 3 次后仍失败，则系统应当标记订单异常并通知运营 |
| REQ-006 | While | **While** 订单处于待支付状态期间，系统应当预占对应库存 |

### 5.2 库存预占与释放模块

| Req ID | 模式 | 需求描述 |
|:---|:---|:---|
| REQ-007 | When | **When** 用户下单时，系统应当创建库存预占记录 |
| REQ-008 | When | **When** 订单支付成功时，系统应当将预占转为实际扣减 |
| REQ-009 | When | **When** 用户取消订单时，系统应当释放预占库存 |
| REQ-010 | When | **When** 订单超时未支付时，系统应当自动释放预占库存 |
| REQ-011 | If | **If** 预占释放时库存数据异常，则系统应当记录日志并告警 |
| REQ-012 | While | **While** 库存处于预占状态期间，其他订单应当看到剩余可用库存 |

### 5.3 库存告警模块

| Req ID | 模式 | 需求描述 |
|:---|:---|:---|
| REQ-013 | When | **When** 商品库存低于安全阈值时，系统应当触发低库存告警 |
| REQ-014 | When | **When** 商品库存降为零时，系统应当标记为 OutOfStock |
| REQ-015 | When | **When** 商品补货入库时，系统应当更新库存数量 |
| REQ-016 | If | **If** 补货后库存仍低于安全阈值，则系统应当保持低库存告警 |
| REQ-017 | While | **While** 商品处于 OutOfStock 状态时，系统应当阻止新的下单操作 |

### 5.4 超卖防护模块

| Req ID | 模式 | 需求描述 |
|:---|:---|:---|
| REQ-018 | When | **When** 订单支付成功时，系统应当再次校验库存是否充足 |
| REQ-019 | If | **If** 支付成功校验时发现库存不足（超卖），则系统应当标记订单异常 |
| REQ-020 | If | **If** 发生超卖，则系统应当自动触发退款流程 |
| REQ-021 | If | **If** 超卖订单超过 10 单/天，则系统应当暂停该商品销售 |
| REQ-022 | Complex | **While** 秒杀活动期间，**When** 订单支付时，系统应当加强超卖校验 |

### 5.5 活动爆单处理模块

| Req ID | 模式 | 需求描述 |
|:---|:---|:---|
| REQ-023 | When | **When** 秒杀活动开始时，系统应当锁定活动库存 |
| REQ-024 | When | **When** 活动库存售罄时，系统应当标记活动结束 |
| REQ-025 | If | **If** 活动并发请求超过系统承载能力，则系统应当启动排队机制 |
| REQ-026 | If | **If** 排队用户超过 10000 人，则系统应当提示"活动火爆，请稍后重试" |
| REQ-027 | If | **If** 活动库存同步出现延迟，则系统应当暂停新订单创建 |
| REQ-028 | While | **While** 活动处于进行中状态期间，系统应当实时监控库存同步状态 |

### 5.6 库存同步延迟处理模块

| Req ID | 模式 | 需求描述 |
|:---|:---|:---|
| REQ-029 | If | **If** 库存同步延迟超过 5 秒，则系统应当标记为同步异常 |
| REQ-030 | If | **If** 同步异常持续超过 30 秒，则系统应当切换为降级模式 |
| REQ-031 | While | **While** 系统处于降级模式期间，系统应当限制下单速率 |
| REQ-032 | When | **When** 库存同步恢复正常时，系统应当解除降级模式 |
| REQ-033 | Complex | **While** 处于降级模式下，**If** 用户尝试下单，则系统应当提示"系统繁忙，请稍后" |

### 6.3 常见误用对比

```
❌ 误用示例（不应出现）：
If 用户选择使用优惠券，则系统应当扣减优惠金额
→ "选择使用优惠券"是正常业务操作，不是异常

✅ 正确使用：
When 用户选择使用优惠券时，系统应当扣减优惠金额
```

```
❌ 误用示例（不应出现）：
If 用户选择候补，则系统应当加入候补队列
→ "选择候补"是正常业务流程

✅ 正确使用：
When 用户选择候补时，系统应当加入候补队列
```
