# 基于 EARS 规范编写 PRD 技能

本技能指导你按照 EARS (Easy Approach to Requirements Syntax) 规范，将业务需求转化为结构化、AI 易读的 PRD 文档。

---

## 一、适用场景判断（第一步）

### ✅ 强烈建议使用 EARS 的场景

- **复杂的业务逻辑与规则**：支付状态流转、订单处理、积分计算、权限判定
- **后台系统交互与 API 规范**：系统间数据流转、接口行为定义
- **状态机相关需求**：订单生命周期、审批流程、任务状态管理
- **异常与容错处理**：超时处理、降级策略、数据一致性保障

### ❌ 不建议使用 EARS 的场景

- **纯前端视觉与 UI/UX 动效**：页面布局、动画效果、交互手势
- **商业目标与市场策略**：用户增长目标、收入指标、营销策略
- **过于早期、发散性的想法**：头脑风暴、概念验证、探索性需求

> **判断口诀**：涉及"条件判断、状态流转、系统具体行为"的逻辑，用 EARS；涉及"好看、好用、商业价值"的描述，不用 EARS。

---

## 二、执行工作流

当用户提供业务需求时，严格按照以下步骤执行：

### Step 0: 需求评估与澄清

**前置判断**：
- 判断用户输入的需求是否适合使用 EARS 语法描述
- 如果需求属于 UI/UX 或商业策略，明确告知用户该技能不适用，并建议其他方法

**输入充分性检查**：
如果用户输入过于模糊（少于 3 句话，或缺少关键业务信息），主动进行澄清式提问：

| 提问维度 | 示例问题 |
|:---|:---|
| **业务实体** | "请问涉及哪些核心业务实体？（如订单、用户、商品）" |
| **系统状态** | "系统有哪些关键状态？各状态之间如何流转？" |
| **触发事件** | "有哪些用户操作或系统事件会触发业务逻辑？" |
| **异常场景** | "有哪些异常情况需要处理？（网络超时、库存不足等）" |
| **约束规则** | "有哪些业务规则或限制条件？（时间限制、数量限制等）" |

**获取足够信息后再进入下一步。**

### Step 1: 需求分析

- 识别业务领域和核心实体
- 确定主要业务流程和分支
- 明确用户角色和系统边界
- **质量检查点**：确认识别了所有可能的状态枚举和关键事件

**🚦 用户确认检查点 #1**
向用户展示当前理解，确认后再继续：
> "我理解您需要构建 **[业务领域]** 系统，核心实体包括 **[实体列表]**。主要业务流程是 **[流程简述]**。请问我的理解正确吗？是否有遗漏或需要调整的地方？"

### Step 2: 提取全局名词

- 列出所有业务术语及其定义
- 明确状态枚举值和取值范围
- 定义关键业务规则和约束
- **质量检查点**：每个状态枚举值都能在后续找到对应的 While 需求

### Step 3: 识别非功能性需求

- 安全性要求
- 性能要求
- 可用性要求
- 合规性要求
- **质量检查点**：NFR 是否覆盖了系统层面的通用约束

### Step 4: 构建 EARS 需求表格

按以下优先级和覆盖度构建需求：

**优先级 1 - 核心主流程（必须覆盖）**
- 使用 When 描述正常流程的事件触发
- 使用 While 描述关键状态下的行为限制

**优先级 2 - 边界与异常（必须覆盖）**
- 使用 **If** 描述所有异常路径和错误处理
- **关键规则**：If 句式**仅用于异常/不期望行为**，如：网络超时、数据缺失、库存不足、服务不可用
- **区分标准**：如果条件是"出了问题"→用 If；如果是正常业务分支→用 When
- 覆盖：空值、越界、超时、服务不可用、数据不一致

**优先级 3 - 复杂组合场景（建议覆盖）**
- 使用 Complex 句式描述多条件组合
- 覆盖：状态+事件、特性+状态、并发冲突

**优先级 4 - 可选特性（按需覆盖）**
- 使用 Where 描述特性开关相关逻辑

**质量检查点**：
- [ ] 正常业务分支使用 When，而非 If
- [ ] 每个状态枚举值都有对应的 While 需求
- [ ] 需求描述中**没有出现技术实现细节**（见下方黑名单）

**🚦 用户确认检查点 #2**
向用户展示已构建的 EARS 需求表格，确认后再继续：
> "我已根据您的需求构建了 **[N] 条功能性需求**，覆盖 **[列举关键 When/While/If 需求]**。请问这些需求是否符合您的预期？是否需要补充或修改任何场景？"

### Step 5: 完整性检查

对照以下清单验证 PRD 完整性：

| 检查项 | 状态 |
|:---|:---:|
| 所有业务术语都有明确定义 | [ ] |
| 每个状态都有对应的 While 需求描述 | [ ] |
| 每个事件都有对应的 When 需求描述 | [ ] |
| 每个异常场景都有对应的 If 需求描述 | [ ] |
| 复杂业务规则使用复合句式描述 | [ ] |
| 没有发现技术实现细节（通过黑名单检查） | [ ] |
| 需求之间没有冲突或矛盾 | [ ] |

**词汇黑名单检查**：
需求描述中**禁止出现**以下词汇：

| 类别 | 禁用词汇示例 |
|:---|:---|
| **数据库** | MySQL、PostgreSQL、MongoDB、Redis、Elasticsearch、表、字段、索引、SQL、事务 |
| **技术栈** | Spring Boot、Django、Node.js、React、Vue、Kafka、RabbitMQ |
| **架构组件** | 微服务、消息队列、负载均衡、容器、K8s、Docker、网关、注册中心 |
| **代码级** | for、while、if、else、循环、递归、API、HTTP、JSON、XML、序列化、反序列化 |
| **缓存** | 缓存、Cache、缓存穿透、缓存雪崩、缓存击穿 |
| **网络协议** | TCP、UDP、WebSocket、gRPC、RESTful、OAuth、JWT（作为实现细节时） |

**If 句式使用正确性检查**：
- ❌ 错误示例（误用）：If 用户选择候补，则系统应当按排队顺序通知
- ✅ 正确示例：**If** 所选日期所有号源已满，则系统应当推荐其他日期
- **判断标准**："患者选择候补"是正常业务流程，应该用 **When** 患者选择候补时...

### Step 6: 组合覆盖度验证（新增）

**目的**：确保所有"状态 × 事件"的组合都被覆盖

**执行步骤**：
1. 列出所有已识别的状态枚举值
2. 列出所有已识别的事件/操作
3. 构建覆盖度矩阵，检查每个有效组合是否有对应需求

**覆盖度矩阵模板**：

```
| 状态 \ 事件 | 创建订单 | 支付 | 取消 | 超时 |
|:---|:---:|:---:|:---:|:---:|
| 待支付 | ✅ REQ-001 | ✅ REQ-005 | ✅ REQ-003 | ✅ REQ-004 |
| 已支付 | - | - | ✅ REQ-007 | - |
| 已取消 | - | - | - | - |

标注说明：
- ✅ 表示该组合已有对应需求
- ❌ 表示该组合缺失需求（需要补充）
- - 表示该组合在业务上不可能发生
```

**覆盖度计算公式**：
- 假设有 N 个状态，M 个事件
- 理论上需要覆盖的组合数 = 所有 (状态, 事件) 中业务上有效的组合
- **质量门禁**：覆盖率应达到 100%（所有有效组合都有需求）

**示例**：
订单有 3 个状态（待支付、已支付、已取消），4 个事件（创建、支付、取消、超时）
- 待支付状态可以响应：创建、支付、取消、超时 → 4 条需求
- 已支付状态可以响应：取消（退款） → 1 条需求
- 已取消状态：所有事件都不响应 → 0 条需求
- 总计应覆盖 5 个有效组合

## 三、EARS 语法速查表

| 模式分类 | 关键字 | EARS 基本句式 | 适用场景 | 使用注意 |
|:---|:---|:---|:---|:---|
| **普遍性需求** (Ubiquitous) | 无特定 | The <系统/模块> shall <响应处理> | 基础属性或通用配置 | 不仅是 NFR，也可描述功能性基础能力 |
| **事件驱动** (Event-driven) | **When** | **When** <触发事件>, the <系统> shall <响应> | 用户操作、系统回调触发 | 正常业务流程的首选句式 |
| **状态驱动** (State-driven) | **While** | **While** <前置状态>, the <系统> shall <响应> | 特定状态下的行为限制 | 每个状态枚举值都应有对应 While |
| **异常/不期望行为** (Unwanted) | **If** | **If** <异常触发>, then the <系统> shall <响应> | 错误处理、容错逻辑 | **仅用于异常场景**，正常分支用 When |
| **可选特性** (Optional) | **Where** | **Where** <使用了某特性>, the <系统> shall <响应> | 特性开关、条件功能 | 描述可选功能的存在条件 |
| **复杂组合** (Complex) | 组合关键字 | **While** <状态>, **When** <事件>, the <系统> shall <响应> | 多重条件组合场景 | 组合条件不超过 3 个为宜 |

### If 句式使用详解（重点）

**核心原则**：If 句式**仅用于描述异常/不期望/错误行为**

| 场景类型 | 示例 | 应使用的句式 |
|:---|:---|:---:|
| **异常/错误** | 网络超时、数据库连接失败、库存不足、权限不足 | **If** |
| **不期望的行为** | 用户输入非法数据、重复提交、并发冲突 | **If** |
| **正常业务分支** | 用户选择 A 方案 vs B 方案、选择候补 vs 直接预约 | **When** |
| **可选操作** | 用户选择使用优惠券、选择加急配送 | **When** |

**常见误用纠正**：

```
❌ 误用 If：
If 患者选择候补，则系统应当按排队顺序通知
→ "选择候补"是正常操作，不是异常

✅ 正确使用 When：
When 患者选择候补时，系统应当按排队顺序通知
```

## 四、核心原则与禁忌

### 1. 业务逻辑优先
- **只描述"做什么"，不描述"怎么做"**
- ❌ 错误：使用 Redis 缓存用户数据，设置 5 分钟过期时间
- ✅ 正确：系统应当缓存用户数据，缓存有效期为 5 分钟

### 2. 消除技术实现细节
禁止出现以下内容：
- 具体技术栈（如：MySQL、Redis、Kafka、Spring Boot）
- 代码实现细节（如：for 循环、SQL 语句、API 路径）
- 架构组件（如：微服务、消息队列、负载均衡）
- 数据库设计（如：表结构、索引、外键关系）

### 3. 边界情况全覆盖
每个业务规则必须考虑：
- **正常路径**：标准流程下的系统响应
- **边界条件**：空值、零值、极限值、临界状态
- **异常情况**：网络超时、服务不可用、数据异常
- **并发场景**：多用户同时操作、竞态条件
- **状态冲突**：不一致状态、过期状态、非法状态组合

### 4. 需求原子化
- 每个需求只描述一个单一、完整的业务行为
- 复杂需求拆分为多个原子需求，使用复合句式关联

## 六、多行业示例库

### 示例 1：电商订单系统

**全局名词定义**：

| 术语 | 定义 | 取值范围 |
|:---|:---|:---|
| **Order** | 用户提交的购买请求 | - |
| **OrderStatus** | 订单当前状态 | [Created, PendingPayment, Paid, Shipped, Delivered, Cancelled, Refunded] |
| **PaymentMethod** | 支付方式 | [Alipay, WeChatPay, CreditCard] |
| **Inventory** | 商品库存 | 非负整数 |

**功能性需求**：

| Req ID | 模式 | 需求描述 |
|:---|:---|:---|
| REQ-001 | When | **When** 用户提交订单时，系统应当创建订单并将状态设置为 Created |
| REQ-002 | If | **If** 订单创建时商品库存不足，则系统应当提示用户并阻止订单创建 |
| REQ-003 | While | **While** 订单处于 PendingPayment 状态期间，系统应当锁定对应商品库存 |
| REQ-004 | If | **If** 订单创建后 30 分钟内未支付，则系统应当自动取消订单并释放库存 |
| REQ-005 | When | **When** 系统收到支付成功通知时，系统应当将订单状态更新为 Paid |
| REQ-006 | If | **If** 支付通知超时未收到，则系统应当标记订单为异常并触发人工审核 |
| REQ-007 | Complex | **While** 订单已发货状态下，**When** 用户申请退货时，系统应当校验退货时限（发货后 7 天内） |
| REQ-008 | Where | **Where** 店铺配置了自动退款功能时，**When** 退货申请审核通过时，系统应当自动发起退款 |

**组合覆盖度矩阵**：

| 状态 \ 事件 | 创建 | 支付成功 | 支付超时 | 取消 | 发货 | 确认收货 | 退货 |
|:---|:---:|:---:|:---:|:---:|:---:|:---:|:---:|
| Created | ✅ REQ-001 | - | - | - | - | - | - |
| PendingPayment | - | ✅ REQ-005 | ✅ REQ-004 | ✅ REQ-003 | - | - | - |
| Paid | - | - | - | - | ✅ REQ-009 | - | - |
| Shipped | - | - | - | - | - | ✅ REQ-010 | ✅ REQ-007 |
| Delivered | - | - | - | - | - | - | ✅ REQ-011 |

### 示例 2：医院预约挂号系统

**全局名词定义**：

| 术语 | 定义 | 取值范围 |
|:---|:---|:---|
| **Appointment** | 患者预约挂号记录 | - |
| **AppointmentStatus** | 预约状态 | [Pending, Confirmed, Completed, Cancelled, NoShow] |
| **TimeSlot** | 号源时段 | 如：09:00-09:30 |
| **DoctorSchedule** | 医生排班 | - |

**功能性需求**：

| Req ID | 模式 | 需求描述 |
|:---|:---|:---|
| REQ-001 | When | **When** 患者选择科室和日期时，系统应当显示可预约的医生列表 |
| REQ-002 | When | **When** 患者选择医生时，系统应当显示该医生的可用时段 |
| REQ-003 | If | **If** 所选时段号源已满，则系统应当提示"号源已满"并推荐其他时段 |
| REQ-004 | When | **When** 患者选择候补时，系统应当将其加入候补队列并按顺序通知 |
| REQ-005 | When | **When** 患者确认预约时，系统应当扣减号源并生成预约记录 |
| REQ-006 | If | **If** 预约时患者已有冲突时段的预约，则系统应当提示时间冲突 |
| REQ-007 | While | **While** 预约处于 Confirmed 状态期间，系统应当保留号源并允许患者取消 |
| REQ-008 | If | **If** 患者未在预约时间前 2 小时取消且未就诊，则系统应当标记为 NoShow |
| REQ-009 | If | **If** 患者 30 天内累计 3 次 NoShow，则系统应当暂停其预约权限 7 天 |
| REQ-010 | When | **When** 医生临时停诊时，系统应当自动取消相关预约并通知患者 |

**关键纠正说明**：
- REQ-004 使用 **When**（不是 If），因为"选择候补"是正常业务流程，不是异常

### 示例 3：物流追踪系统

**全局名词定义**：

| 术语 | 定义 | 取值范围 |
|:---|:---|:---|

## 八、常见问题解答 (FAQ)

**Q1: 用户说"帮我设计一个社交 APP 的首页，要好看、有动效"，应该怎么办？**
A: 这是 UI/UX 设计需求，不适合使用 EARS。应回复："该需求涉及前端视觉和交互设计，不建议使用 EARS 语法描述。建议提供线框图、交互原型或设计稿来表达这类需求。"

**Q2: "用户选择使用优惠券"应该用 When 还是 If？**
A: 用 **When**。"选择使用优惠券"是用户的正常操作，不是异常场景。

**Q3: 如何区分"正常业务分支"和"异常"？**
A: 判断标准：这个条件是否代表"出了问题"或"出现了错误"？
- 是 → 用 If（如：库存不足、网络超时、权限不足）
- 否 → 用 When（如：用户选择 A 方案、选择候补、使用优惠券）

**Q4: 组合覆盖度 100% 是什么意思？**
A: 指所有"状态 × 事件"在业务上有效的组合，都应有对应的需求描述。例如"已取消的订单"通常不需要响应"支付"事件，这种组合标记为"-"（不可能发生），不需要写需求。
