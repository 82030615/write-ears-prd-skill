# 使用 EARS 编写标准且 AI 易读的 PRD 指南

## 1. 什么是 EARS (Easy Approach to Requirements Syntax)?

**EARS** 是一种通过少量关键字和简单规则来约束自然语言的结构化需求表达方法。它的核心目的是消除自然语言中的歧义，统一需求的编写格式。
在 AI（尤其是大语言模型 LLM）时代，EARS 的高度结构化语法简直是为 AI 量身定制的：**它将模糊的业务需求直接转化为类似于代码逻辑（If/When/While/Then）的清晰指令**，从而大幅提高 AI 生成代码、测试用例的准确率。

---

## 2. 快速判断：什么时候该用 EARS？

并非所有文档都适合用 EARS。你可以通过以下准则快速判断：

### ✅ 强烈建议使用 EARS 的场景（适合 AI 解析开发）：
*   **复杂的业务逻辑与规则**：例如支付状态流转、订单处理、积分计算。
*   **后台系统交互与 API 规范**：系统 A 如何根据系统 B 的回调做出响应。
*   **状态机相关需求**：例如“登录中”、“已断开”、“未实名”各种状态下系统的不同表现。
*   **软硬件交互/物联网应用**：按钮按下、传感器报警时系统的行为。
*   **异常与容错处理**：网络中断、超时、数据异常时的逻辑分支。

### ❌ 不建议使用 EARS 的场景：
*   **纯前端视觉与 UI/UX 动效**：“按钮悬浮时应该有一个平滑的发光动画并伴随缩放”。（AI 会更容易理解 CSS/Figma，EARS 不擅长描述视觉感官）。
*   **商业目标与市场策略**：“我们要提高 20% 的用户留存率”。（这属于 OKR/商业计划，不属于系统行为要求）。
*   **过于早期、发散性的想法**：还没确定系统具体边界要做什么的时候。

**💡 总结语**：**涉及“条件判断”、“状态流转”、“系统具体行为”的“硬核”逻辑，用 EARS；涉及“好看”、“好用”、“商业价值”的“软性”描述，不用 EARS。**

## 3. EARS 的 6 大标准语法（AI 的完美 Parser 规则）

将这些句式融入你的 PRD，AI 阅读时能极大概率准确将其翻译成对应的代码逻辑。

| 模式分类 | 关键字 | EARS 基本句式 (中英文对应) | 对应 AI / 代码逻辑的理解 | 示例 |
| :--- | :--- | :--- | :--- | :--- |
| **1. 普遍性需求** (Ubiquitous) | 无特定 | The **<系统/模块>** shall **<响应处理>**. | 基础属性或通用配置 (Config / Base) | 系统应当限制用户上传文件在 50MB 以内。 |
| **2. 事件驱动** (Event-driven) | **When** | **When** <触发事件>, the **<系统>** shall **<响应>**. | 事件回调: `onEvent() { doSomething() }` | **当** 用户点击“提交”时，系统应当验证所有必填字段。 |
| **3. 状态驱动** (State-driven) | **While** | **While** <前置状态>, the **<系统>** shall **<响应>**. | 状态校验: `if (state == X) { allow() }` | **在** 用户处于“未实名”状态**期间**，系统应当隐藏“立即提现”入口。 |
| **4. 异常/不期望行为** (Unwanted) | **If** | **If** <异常触发>, then the **<系统>** shall **<响应>**. | 异常捕获: `try/catch` 或 `if (error)` | **遇到** 数据库连接超时，系统应当展示重试页面并输出日志。 |
| **5. 可选特性** (Optional) | **Where** | **Where** <使用了某特性>, the **<系统>** shall **<响应>**. | 特性开关: Feature Flag | **在** 配置了指纹模块的设备上，系统应当优先提示指纹解锁。 |
| **6. 复杂组合** (Complex) | 组合关键字 | **While** <状态>, **When** <事件>, the **<系统>** shall **<响应>**. | 多重组合条件逻辑 | **在** 飞行模式下，**当** 用户尝试发起网络请求时，系统应当以弹窗报错。 |

## 4. 标准、AI 易读的 PRD 模板结构示范

为了让 AI 工具（如 GitHub Copilot, Cursor 或各类开发 Agent）能够吃透你的 PRD 快速生成代码，建议使用以下结构编写：

### 4.1 全局名次定义 (Glossary) - 极其重要
*(给 AI 提供明确的实体字典，防止大模型自己“幻觉”编造名词)*
*   **User**: 指已完成手机号注册并处于 Active 状态的用户。
*   **Order_Status**: 订单系统状态枚举值，仅包含 `[Pending, Paid, Cancelled, Refunded]`。

### 4.2 非功能性需求 (Non-Functional Requirements)
*   **NFR-01 (Ubiquitous)**: 订单系统所有数据交互必须采用加密传输。

### 4.3 核心功能流转 (EARS Requirements Table)
*(大语言模型非常擅长解析 Markdown 表格，并直接将其转换为代码逻辑及测试用例)*

| Req ID | 模式 | 需求描述 (按照 EARS 规范) |
| :--- | :--- | :--- |
| REQ-001 | 状态驱动 | **在(While)** 用户处于黑名单状态期间，系统将拦截该用户的所有发帖请求，并返回 403。 |
| REQ-002 | 事件驱动 | **当(When)** 订单系统接收到来自支付宝的付款成功回调时，系统应更新订单状态至 Paid。 |
| REQ-003 | 异常处理 | **如果(If)** 第三方发短信 API 连续 3 次返回失败，系统应触发告警并自动切换到备用短信通道。 |
| REQ-004 | 复杂结构 | **在(While)** 购物车非空期间，**当(When)** 购物车内某商品全局库存为 0 时，系统应将该商品标记为“已失效”且不可勾选。 |

## 5. 为什么用 EARS 能极大提升 AI 辅助开发效率？

1.  **秒变测试驱动 (TDD)**：EARS 句式可以极其轻松地 1:1 被 AI 转换为自动化测试用例。例如 `While (Arrange 条件)` + `When (Act 操作)` + `Shall (Assert 断言)`。
2.  **消除代词造成的上下文迷失**：自然语言里大量使用“这个”、“它”、“然后这么处理”会让 AI 迷失操作主体。EARS 严格要求 `the <系统>`，让 AI 明确动作执行者和作用域。
3.  **倒逼业务提供异常分支补全**：当你习惯使用 `When` 的同时，你会自然逼迫自己思考各种 `If(异常)`（如网络很卡怎么办？API无规律响应怎么办？），从而让交付给 AI 的需求天生就是一张没有任何漏洞的高质量大网。
